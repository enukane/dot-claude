# TDD（テスト駆動開発）ポリシー

> コーディングタスクにおいて Claude Code が従うべきTDDガイドラインです。

---

## 基本原則

**CRITICAL**: TDDの本質は「仕様をテストコードとして記述すること」である。

## TDDサイクル

1. **RED**: 失敗するテストを書く
   - 仕様の詳細（サイズ、色、フォント等）を具体的な値で検証
   - エッジケース・異常系を最初から計画
2. **GREEN**: 最小限のコードでテストをパス
3. **REFACTOR**: コードを改善（定数化、命名、責務の分離）
4. **FEEDBACK**: 設計への気づきを反映

## テスト作成の必須要件

### 正常系
- [ ] 具体的な値を検証（例: `expect(style.width).toBe(48)`）
- [ ] サイズ、色、フォントなど仕様の詳細をテスト

### エッジケース
- [ ] 空の値（空文字列、undefined、null）
- [ ] 非常に長い値（1000文字など）
- [ ] 矛盾する状態の組み合わせ（loading + disabled など）
- [ ] デフォルト値の確認
- [ ] 連続操作の挙動

**重要**: 「動作する」だけでなく、「どのように動作するか」を検証する。
